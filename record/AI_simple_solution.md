# 第一阶段：探索 + 出方案

> 目标：将"Web 前端美学质量"这个主观模糊概念，拆解为可操作、可复现、可规模化的数据管线方案。

---

## 1. 美学质量的可操作维度拆解

将"美学好不好"拆为 **4 层、12 个可判定的维度**：

### L1 — 基础可用性（硬伤检测，全部可机判）
| 维度 | 定义 | 判定方式 |
|------|------|----------|
| **布局完整性** | 无明显溢出、重叠、断裂；flex/grid 正常渲染 | 机判（Puppeteer 截图 + 布局检测） |
| **可读性** | 对比度 ≥ WCAG AA（4.5:1）；字号 ≥ 12px；行高 ≥ 1.4 | 机判（CSS 提取 + 对比度计算） |
| **响应式基础** | 在 375px / 1440px 宽度下均无水平滚动条 | 机判（双视口截图对比） |
| **加载与性能** | 无白屏、无闪烁；FCP < 1.5s | 机判（Lighthouse / Performance API） |

### L2 — 视觉协调性（规则可编码，大部分可机判）
| 维度 | 定义 | 判定方式 |
|------|------|----------|
| **色彩和谐度** | 配色是否遵循色彩理论（互补/类比/三角色方案）；主色 ≤ 3 种 | 机判（提取色板 → 色相分布分析） |
| **间距一致性** | 同类元素间距方差小（8px 网格系统）；padding/margin 有规律 | 机判（DOM 间距提取 + 统计分析） |
| **字体层级** | 页面字体族 ≤ 2；字号层级清晰（h1-h6 递减）；无"全是同一字号" | 机判（CSS 字体属性提取） |
| **视觉重心** | 有明确的视觉焦点（首屏有主导元素）；无"到处都在抢注意力" | **人判为主**，机判辅助（眼动热力图模拟） |

### L3 — 设计感（需要美学知识，人判为主）
| 维度 | 定义 | 判定方式 |
|------|------|----------|
| **留白运用** | 是否恰当使用负空间；不拥挤也不空洞 | **人判** |
| **设计原创性** | 是否跳出"AI slop"模板感——千篇一律的蓝紫渐变 + 圆角卡片 + emoji icon | **人判**（可机判辅助：与已知 slop 模板的相似度检测） |
| **品牌感/调性** | 页面是否有统一的视觉语言、氛围；不只是组件堆砌 | **人判** |

### L4 — 交互与动效（锦上添花）
| 维度 | 定义 | 判定方式 |
|------|------|----------|
| **微交互自然度** | hover/click 过渡是否流畅自然；动画时间曲线合理 | 人判为主，机判辅助（动画帧分析） |

## 2. 机判 vs 人判的边界

```
┌─────────────────────────────────────────────────────┐
│  纯机判              │  机判+人校验      │  纯人判    │
│  (L1 + L2 大部分)    │  (L2 视觉重心等)  │  (L3 + L4) │
│──────────────────────│──────────────────│────────────│
│  通过率 ~70%         │  通过率 ~15%      │ ~15%       │
│  成本: 几乎为零       │  成本: 中         │  成本: 高   │
└─────────────────────────────────────────────────────┘
```

### 边界划分原则：
- **能写成确定性规则的 → 机判**（对比度、溢出检测、字号分布）
- **规则模糊但异常值可检出的 → 机判筛 + 人确认**（机判标"可疑"，人做二分类）
- **需要文化/审美/时代语境的 → 人判**（留白、原创性、调性）
- **争议项不强行机判**：宁可标记 `NEED_HUMAN` 也不让算法硬打分

### 人判环节的设计：
- 采用**偏好对（A vs B）**而非绝对打分——人对"哪个更好"的判断远比对"这个打几分"可靠
- 每人每次判 10-20 对，控制疲劳
- 每对至少 3 人独立判，用多数票 + Fleiss' Kappa 衡量一致性
- 一致性 < 0.4 的样本标记为"高争议"单独分析，不直接进入训练集

## 3. 数据格式选择：混合模式

### 推荐：**多维 rubric 判据 + 偏好对**，而非单一分数

原因：
- 单一分数（"7.2 分"）丢失太多信息——两个 7 分的页面可能一个配色好布局烂，一个反之
- 偏好对在 human labeling 中 reliability 最高
- rubric 让评分可解释、可复现

## 4. Ground Truth 来源——防"LLM 循环"

### 核心原则：**真值不能来自另一个 LLM 的"打分"**

### Ground truth 分层来源：

| 层级 | 来源 | 用途 |
|------|------|------|
| **L1 硬伤** | 确定性规则（WCAG 公式、DOM 解析） | 可直接作为 ground truth |
| **L2 协调性** | 设计系统规范 + 人类设计师校准过的阈值 | 机判分数需与人类判断做校准后使用 |
| **L3 设计感** | **多人标注的偏好对**（≥3人，来自不同背景） | 作为训练/评估的真值 |
| **锚点样本** | **精选的"标杆"页面**（Awwwards、Stripe、Linear 等公认好设计）作为正锚点；典型的 AI slop 作为负锚点 | 用于校准和 embedding 空间对齐 |

### 防循环的具体措施：

1. **禁止 LLM-as-judge 做最终打分**：LLM 只能用于（a）解释它为什么生成这样的代码（内省），（b）辅助机判的规则编写，但不能直接输出"这个页面 8/10"
2. **人类标签是 L3/L4 的唯一真值来源**：即使是小样本（50-100 条），也比大样本 LLM 打分可靠
3. **锚点校准法**：每个 human 判例中混入 2-3 个已知锚点（已多人标注过的），用于检测标注者偏倚和疲劳
4. **LLM 可用于"找茬"而非"打分"**：让 LLM 指出"这个页面哪里可能不好"（具体缺陷），但不让它给整体分数。具体缺陷是可验证的（"这个按钮对比度不足"→ 机判可确认）

## 5. 防多样性坍缩

### 多样性坍缩的来源：
- 用同一个基础 prompt 生成大量 UI → prompt 层面缺乏变异
- 评分标准偏向某种"安全"风格 → 筛选压力导致同质化
- 人类标注者偏好趋同 → label bias

### 应对策略：

| 策略 | 具体做法 |
|------|----------|
| **Prompt 多样性注入** | 建立 prompt 变异矩阵：风格（极简/新粗野主义/玻璃态/复古 web 等）× 行业（SaaS/电商/作品集/医疗等）× 色调（暖/冷/中性）× 复杂度（单页/仪表盘/落地页）。系统性地从每个格子采样 |
| **"反 slop"约束** | 在 prompt 中显式加入反模板指令："不要使用蓝紫渐变作为背景""不要使用 emoji 作为 hero icon""不要使用常见的 3 列 pricing 卡片布局" |
| **多样性指标监控** | 在生成的 HTML 上做 embedding（CLIP image embedding），监控样本间的 pairwise cosine similarity 分布。相似度中位数 > 0.85 时触发多样性告警 |
| **Human-in-the-loop 的多样性** | 标注者来自不同背景（设计师 / 前端开发 / 普通用户），减少单一审美偏好 |
| **保留"奇怪但有趣"的样本** | 极端值不自动丢弃——设计创新往往看起来"奇怪"。标记但保留 |

## 7. 第一阶段探索路径

### Step 1: 感性地建立"好/坏"的直觉（~1h）
- 用多种 prompt 在不同模型上生成一批 UI（目标：50-100 个）
- 人工快速浏览，贴标签："好""还行""AI slop""有硬伤"
- 输出：一个初始的**坏味道清单** + **好样本锚点集**

### Step 2: 形式化判据（~1h）
- 基于 Step 1 的发现，细化 rubric 的每个维度
- 为每个维度写"1 分长什么样 / 3 分长什么样 / 5 分长什么样"
- 找 3 个锚点样本验证 rubric 是否能用（独立判、比结果）

### Step 3: 机判模块 MVP（~2h）
- 先做对比度检测 + 色板分析 + 布局溢出检测
- 在 Step 1 的 50-100 个样本上跑，看机判结果和人工"好/坏"标签的相关性

### Step 4: 人判流程设计（~0.5h）
- 设计偏好对标注任务的 UI（可以是最简单的 HTML form）
- 设计锚点混入策略
- 设计标注者一致性计算方法

### Step 5: 方案定稿（~0.5h）
- 综合以上，写出最终方案文档
- 明确：哪些写代码、哪些靠 prompt、哪些靠人

## 8. 关键风险 & 已知局限（预判）

1. **"设计好"本身的文化依赖性**：日式极简和欧美 brutalism 在同一 rubric 下可能评分迥异。解决方案：rubric 按风格分轨，或标注风格标签做分层分析。
2. **机判维度的覆盖面有限**：L1+L2 大约覆盖"不丑"的底线，但判断"多美"还是依赖人。这意味着自动化评分可能只有"筛掉差的"的能力，没有"排出好的"的能力。
3. **人判的成本与规模瓶颈**：这是整个管线最大的瓶颈。初始阶段接受小样本（50-100 人判），验证方案后再考虑 scaling。
4. **偏好对数据的稀疏性**：n 个样本做两两比较需要 O(n²) 对，不可扩展。实际会用 Elo rating 系统 + 主动学习选最有信息量的对。
